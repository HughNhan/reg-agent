# Phase 1: QUADS Resource Broker Module

## Purpose

**Resource Broker** for Red Hat performance/scale lab infrastructure. Handles complete resource lifecycle: discovery, allocation, importation, validation, and deallocation.

**Architectural Note**: This module is intentionally a **Resource Broker**, not just an allocator. It manages all QUADS API interactions and provides a clean abstraction for downstream phases. See `/ARCHITECTURE-QUADS-RESOURCE-BROKER.md` for design rationale.

## Module Scope

**Focus**: Complete QUADS resource lifecycle (broker)

**Current Capabilities**:
- Allocate new resources
- Import existing allocations
- Validate assignment status
- Discover available resources
- Deallocate/terminate assignments

**Planned Enhancement - AI-Driven Attribute Matching**:
- AI layer specifies required hardware attributes (e.g., "BlueField-3 DPU", "ConnectX-7 NIC")
- QUADS broker translates requirements → QUADS API queries
- Semantic resource matching (by capability, not just model)
- Example: "Test DPU performance" → Broker finds servers with BF-3 DPUs
- See `ARCHITECTURE-QUADS-RESOURCE-BROKER.md` for detailed design

**Input**: Configuration from `vars/config.env`
**Output**: State in `vars/state.env` (CLOUD_NAME, ASSIGNMENT_ID, LAB)
**API Boundary**: Only this module talks to QUADS API
**Next Phase**: Jetlag cluster deployment (Phase 2)

## Key Script: `quads-ssm-reserve.sh`

**Location**: `modules/quads/quads-ssm-reserve.sh`
**Purpose**: Request machines from QUADS API via ansible-quads-ssm
**Called by**: `Makefile` target `test-quads` or `deploy-full`

### Dependencies Checked

The script uses `../lib/check-dependencies.sh` to validate:

1. **Repository**: ansible-quads-ssm exists at `repos/ansible-quads-ssm/`
2. **Configuration variables** (from `vars/config.env`):
   - QUADS_API_SERVER
   - QUADS_USERNAME
   - QUADS_PASSWORD
   - LAB (scalelab or performancelab)
   - NUM_HOSTS (number of machines)
   - PREFERRED_MODEL (hardware model, e.g., r750)
   - WORKLOAD_NAME (reservation identifier)
3. **Commands**: ansible-playbook, jq, curl
4. **Network**: Access to QUADS API server

### What It Does

1. **Dependency Check**: Validates all prerequisites
2. **Generate Config**: Creates `repos/ansible-quads-ssm/quads_config.yml`:
   ```yaml
   quads_api_server: "http://quads.example.com:8080"
   quads_username: "your-username"
   quads_user_domain: "redhat.com"
   quads_password: "your-password"
   preferred_models:
     - "r750"
     - "r740xd"
   ```
3. **Run Ansible**: Executes `quads_self_schedule.yml` playbook
4. **Parse Output**: Extracts CLOUD_NAME and ASSIGNMENT_ID from output
5. **Wait for Validation**: Polls QUADS API until assignment is validated
6. **Save State**: Writes to `vars/state.env`:
   ```bash
   CLOUD_NAME=cloud42
   ASSIGNMENT_ID=12345
   QUADS_METHOD=quads-ssm
   LAB=scalelab
   ```

### Configuration Variables

**Required** (from `vars/config.env`):
- `QUADS_API_SERVER`: QUADS API URL (e.g., http://quads.scalelab.redhat.com:8080)
- `QUADS_USERNAME`: Your username
- `QUADS_PASSWORD`: Your QUADS password
- `LAB`: scalelab or performancelab
- `NUM_HOSTS`: Number of machines to reserve (e.g., 6)
- `PREFERRED_MODEL`: Hardware model (e.g., r750, r740xd, or comma-separated list)
- `WORKLOAD_NAME`: Identifier for reservation (e.g., reg-agent-20260609)

**Optional**:
- `QUADS_USER_DOMAIN`: Defaults to "redhat.com"
- `WIPE_DISKS`: yes/no, defaults to "yes" (set to "no" for testing allocate/deallocate cycles)
- `SHORT_DESCRIPTION`: Short description for the assignment (defaults to WORKLOAD_NAME)
- `QUADS_API_TOKEN`: SSO API token (preferred over password authentication)

### Output State

Written to `vars/state.env`:
- `CLOUD_NAME`: Allocated cloud identifier (e.g., cloud42)
- `ASSIGNMENT_ID`: QUADS assignment ID for tracking
- `QUADS_METHOD`: Always "quads-ssm" for this script
- `LAB`: Echoed from config

## Key Script: `quads-ssm-import.sh`

**Location**: `modules/quads/quads-ssm-import.sh`
**Purpose**: Import existing QUADS allocation into reg-agent workspace
**Called by**: `Makefile` target `import`

### What It Does

1. **Dependency Check**: Validates API access and authentication
2. **Input Resolution**: Gets CLOUD_NAME and LAB from command-line, config file, or interactive prompt
3. **Authenticate**: Gets API token via SSO token or password
4. **Validate Cloud**: Queries QUADS API to verify cloud exists
5. **Find Assignment**: Locates active assignment for the specified cloud
6. **Retrieve Details**: Gets assignment metadata (owner, description, host count, etc.)
7. **Check Conflicts**: Warns if workspace already has a different allocation
8. **Save State**: Writes allocation info to both module state and global state

### Required Variables

From `vars/config.env`:
- `QUADS_API_SERVER`: QUADS API URL
- `QUADS_USERNAME`: Your username
- Either `QUADS_API_TOKEN` or `QUADS_PASSWORD`: Authentication credentials
- `LAB`: Lab location (scalelab or performancelab) - can be provided via command-line

### Input Methods

**Priority 1 - Command-line arguments** (recommended):
```bash
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab
```

**Priority 2 - Interactive configuration**:
```bash
make -C modules/quads init
# Choose option 3: Import existing allocation
make -C modules/quads import
```

**Priority 3 - Global config** (`vars/config.env`):
```bash
CLOUD_NAME=cloud23
LAB=scalelab
```

### Output State

Written to both `modules/quads/generated/state/current.env` and `vars/state.env`:
```bash
CLOUD_NAME=cloud23
ASSIGNMENT_ID=12345
QUADS_METHOD=quads-ssm
LAB=scalelab
DEPLOYMENT_METHOD=imported
QUADS_IMPORT_COMPLETED=true
QUADS_IMPORT_TIMESTAMP=20260610_123456
ASSIGNMENT_OWNER=username
ASSIGNMENT_DESC=description
NUM_HOSTS=6
WIPE_DISKS=no
```

### Validation Checks

The import script performs these validations:

1. **Cloud exists**: Verifies the cloud name is valid in QUADS
2. **Active assignment**: Confirms there's an active assignment for the cloud
3. **Assignment status**: Checks if assignment is active (warns if not validated)
4. **Workspace conflicts**: Warns if workspace already manages a different allocation
5. **Authentication**: Verifies QUADS API credentials are valid

### Usage

```bash
# Import by cloud name and lab
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab

# Or from project root
cd /path/to/reg-agent
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab

# Validate after import
make -C modules/quads validate
```

### Exit Codes

- `0`: Import successful, state saved
- `1`: Import failed (cloud not found, no active assignment, authentication failed, etc.)

### Common Use Cases

**Scenario 1: Resume work on existing allocation**
```bash
# You allocated cloud23 manually via QUADS UI or another tool
# Now you want to use it with reg-agent
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab
make test-jetlag  # Continue with Phase 2
```

**Scenario 2: Share allocation between team members**
```bash
# Team member A allocated cloud15
# Team member B wants to import it
make -C modules/quads import CLOUD_NAME=cloud15 LAB=scalelab
make validate
```

**Scenario 3: Recover from lost state**
```bash
# You lost vars/state.env but allocation still exists
# Re-import to restore state
make -C modules/quads import CLOUD_NAME=cloud42 LAB=scalelab
```

## Key Script: `quads-ssm-validate.sh`

**Location**: `modules/quads/quads-ssm-validate.sh`
**Purpose**: Validate existing QUADS assignment status
**Called by**: `Makefile` target `validate-quads`

### What It Does

1. **Dependency Check**: Validates API access and authentication
2. **Query Assignment**: Retrieves assignment details from QUADS API
3. **Check Status**: Verifies assignment is active and validated
4. **List Hosts**: Shows all hosts assigned to the cloud
5. **Report Results**: Displays comprehensive status information

### Output Information

- Assignment ID and cloud name
- Description and owner
- Validated status (true/false)
- Active status (true/false)
- Number of hosts and their details
- Wipe and QINQ settings
- Lab location

### Exit Codes

- `0`: Assignment is valid, active, and ready
- `1`: Assignment is inactive or terminated
- `2`: Assignment exists but not validated yet

### Usage

```bash
make validate-quads
```

## Key Script: `quads-ssm-deallocate.sh`

**Location**: `modules/quads/quads-ssm-deallocate.sh`
**Purpose**: Terminate a QUADS assignment and release resources
**Called by**: `Makefile` target `deallocate-quads`

### What It Does

1. **Dependency Check**: Validates API access and state
2. **Load State**: Reads ASSIGNMENT_ID and CLOUD_NAME from `vars/state.env`
3. **Authenticate**: Gets API token (via SSO token or password)
4. **Check Status**: Queries assignment before terminating
5. **Confirm**: Prompts user for confirmation
6. **Terminate**: Calls QUADS API to terminate assignment
7. **Clean State**: Removes `vars/state.env` after successful termination

### Required State

Reads from `vars/state.env`:
- `ASSIGNMENT_ID`: The assignment to terminate
- `CLOUD_NAME`: Cloud identifier
- `LAB`: Lab location

### Usage

```bash
make deallocate-quads
```

**Warning**: This action cannot be undone. The script will prompt for confirmation before proceeding.

## QUADS Management Workflow

### Workflow 1: Allocate New Cluster

```bash
# 1. Configure reg-agent
make configure

# 2. Allocate a new cluster
make test-quads

# 3. Check if allocation is ready
make validate-quads

# 4. When done, release the cluster
make deallocate-quads
```

### Workflow 2: Import Existing Allocation

```bash
# 1. Configure QUADS credentials (if not already done)
make configure

# 2. Import existing allocation by cloud name
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab

# 3. Validate the imported allocation
make validate-quads

# 4. Continue with next phase (Jetlag)
make test-jetlag

# 5. When done, optionally deallocate
make deallocate-quads
```

### Testing with No-Wipe Flag

For rapid testing (allocate → deallocate → allocate cycles):

```bash
# Set WIPE_DISKS=no in vars/config.env
echo "WIPE_DISKS=no" >> vars/config.env

# Allocate cluster
make test-quads

# Work with cluster...

# Deallocate
make deallocate-quads

# Re-allocate (much faster without disk wipe)
make test-quads
```

### Cluster Metadata

Each QUADS allocation includes:
- **Lab**: `scalelab` or `performancelab`
- **Cloud Name**: e.g., `cloud04`, `cloud42`
- **Short Description**: Human-readable description (e.g., "dataplane development")
- **Wipe Flag**: Whether to wipe/rebuild disks
- **QINQ**: VLAN mode (0 or 1)

## Common Issues

### Issue 1: "Repository missing: ansible-quads-ssm"
**Fix**: Run `./bootstrap.sh` to clone dependencies

###Issue 2: "Variable not set: QUADS_API_SERVER"
**Fix**: Run `make configure` or manually edit `vars/config.env`

### Issue 3: "Could not extract cloud_name from QUADS output"
**Cause**: QUADS API returned unexpected format
**Debug**: Check `/tmp/quads_output.log` for actual API response
**Fix**: Verify QUADS API is accessible and credentials are correct

### Issue 4: "Validation timeout reached"
**Cause**: QUADS assignment not validated within 10 minutes
**Impact**: Non-fatal - script continues anyway
**Note**: Assignment may still be valid, just not marked as validated yet

### Issue 5: "Cloud not found" (during import)
**Cause**: Cloud name doesn't exist in QUADS or typo in name
**Fix**:
```bash
# Check available clouds
make -C modules/quads show-available

# Or list your assignments via API
curl -s -k -H "Authorization: Bearer $QUADS_TOKEN" \
  "https://${QUADS_API_SERVER}/api/v3/assignments?owner=${QUADS_USERNAME}" | jq
```

### Issue 6: "No active assignment found" (during import)
**Cause**: Cloud exists but has no active assignment
**Fix**: The allocation may have expired or been terminated. Check QUADS UI or allocate a new cloud.

## Using Existing Allocations

There are two ways to use an existing QUADS allocation:

### Method 1: Import Script (Recommended)

Use the dedicated import script to validate and import existing allocations:

```bash
make -C modules/quads import CLOUD_NAME=cloud23 LAB=scalelab
```

This method:
- Validates the cloud exists and has an active assignment
- Retrieves full assignment details from QUADS API
- Checks for workspace conflicts
- Saves complete state information

### Method 2: Config Flag (Legacy)

Set in `vars/config.env`:
```bash
USE_EXISTING_ASSIGNMENT=true
CLOUD_NAME=cloud42
LAB=scalelab
```

Then run:
```bash
make test-quads
```

This method:
- Skips allocation and just saves the cloud name
- Does not validate the cloud exists
- Does not retrieve assignment metadata
- Useful for quick testing only

## ansible-quads-ssm Details

**Repository**: https://github.com/quadsproject/ansible-quads-ssm
**Purpose**: Ansible automation for QUADS self-service scheduling
**Key Playbook**: `quads_self_schedule.yml`

**Input** (via -e flags):
- `workload_name`: Identifier for your reservation
- `num_hosts`: Number of machines
- `wipe`: true/false for disk wiping

**Output** (JSON response):
```json
{
  "cloud_name": "cloud42",

  "validated": false
}
```

## Development Tips

### Adding New Hardware Models
1. Edit `vars/config.env`
2. Set `PREFERRED_MODEL=r750,r740xd,r650` (comma-separated)
3. The script converts to YAML list format

### Debugging API Calls
1. Check `/tmp/quads_output.log` for full ansible output
2. Use `curl -sk "https://quads-host/api/v3/assignments/$ASSIGNMENT_ID"` to query assignment status

### Modifying Dependency Checks
Edit the "Check dependencies" section in `quads-ssm-reserve.sh`:
```bash
check_repo "ansible-quads-ssm"
check_var "New variable" "NEW_VAR"
check_command "New command" "newcmd"
```
